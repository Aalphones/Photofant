# FINDINGS — P22 Knowledge Engine

> Getaggte Erkenntnisse während der Umsetzung. Format: `- [ ] → Phase N: <Erkenntnis / Entscheidung / Abweichung>`. Mechanik: `mode-implementing`. Bestätigte Architektur-Erkenntnisse wandern in ein ADR, dauerhafte Regeln in die Conventions.

- [x] → alle Phasen (Kontrakt): **Settings-Keys sind snake_case**, nicht camelCase. Real angelegt: `knowledge.vault_path` (Default `None` → `<data_root>/knowledge`), `knowledge.default_domain` (Default `"Movies"`) in `settings.py`. Der Plan hatte `vaultPath`/`defaultDomain` notiert — korrigiert, weil die ganze `settings.py` snake_case ist (auch die ADR-Prosa nutzt teils camelCase, der echte Key aber snake_case, z.B. `rerank.candidate_pool_size`). README-Kontrakt nachgezogen.
- [x] → alle Phasen (Kontrakt): **ADR-Nummer ist 025**, nicht 010. Die Plan-Reservierung „nächste freie = 010" beruhte auf einem veralteten `docs/decisions/`-Stand; 010 (Bildklassifizierung) und 011 (Galerie-Virtual-Scroll) sind längst belegt, ADRs laufen bis 024. Vault-ADR = `025-knowledge-vault-markdown-wahrheit.md`. Der Jobs-ADR (P24) bekommt dort die dann freie Nummer.
- [x] → Phase 2 (Cache): **Owner-Priorität + Overwrite-Regel leben in `schema.py`** (`Owner`-Enum, `owner_priority`, `owner_can_overwrite`; MVP: gleiche oder höhere Priorität darf überschreiben, niedrigere wird abgelehnt). Der Cache-Schreibpfad muss das respektieren, nicht neu erfinden. — Eingearbeitet: `EntityRepository.upsert_from_vault` ist wie `vault.save_entity` reines I/O (voller Zeilen-Ersatz aus einer bereits vom Service geprüften `Entity`, kein Feld-Merge, keine eigene Ownership-Logik). Owner-Werte werden nur als `str` gespeichert, nicht neu bewertet — die Entscheidung bleibt beim `KnowledgeService` (Phase 3), genau wie unten für `vault.save_entity` festgehalten.
- [x] → Phase 3 (Service): **`EntityRepository`/`RelationshipRepository` sind lesend planbar für `find_entity`/`get_lore`.** `find_by_alias(alias)` und `search(query, type?, domain?)` liefern beide `list[KnowledgeEntity]` (keine Ambiguitäts-Auflösung eingebaut) — Eingearbeitet: `find_entity(ref)` löst mehrdeutige Alias-Treffer **nicht** still auf (erstes Ergebnis wäre eine stille Falschauswahl in einer Markdown-first-Wissensbasis), sondern wirft `AmbiguousEntityError` (REST → 409) — der Aufrufer (z.B. UI) entscheidet. `RelationshipRepository.referencing(target_id)` liefert eingehende Kanten für Graph-Traversal (`get_lore`, spätere Empfehlungen) — in Phase 3 noch ungenutzt (`get_lore` liest nur ausgehende Kanten via `for_entity`), Kandidat für P25/P26.
- [x] → Phase 3 (Service): **Ownership ist entity-weit, nicht pro Feld.** `schema.Entity.owner` ist ein einzelnes Feld — `update_entity`/`create_relationship`/`remove_relationship` prüfen `owner_can_overwrite(writer, entity.owner)` einmal für den gesamten Schreibzugriff; bei Erfolg wird der Schreiber der neue `owner` der ganzen Entity. Passt zur Kontrakt-Formulierung „Schreibzugriff mit niedrigerer Owner-Priorität überschreibt keinen höheren Wert" (Singular „Wert", kein Per-Feld-Konzept im Schema).
- [x] → Phase 3 (Service): **`update_entity`-Patch lehnt `id`/`type`/`domain` explizit ab** (422), statt sie still zu ignorieren — alle drei sind Teil des unveränderlichen Pfad-Kontrakts (`id` fix laut README-Risiken, `type`/`domain` bestimmen den Vault-Ordner). Ein Titel-/Alias-Update braucht **keinen** Datei-Move, da `entity_path()` nur von `id`/`type` abhängt, nicht von `title`.
- [x] → Phase 3 (REST): **`owner` ist ein echtes, optionales Request-Feld** (Default `"user"`) auf `POST/PATCH/POST relationships/DELETE relationships`, nicht serverseitig hart auf `user` fixiert. Grund: Die Plan-Smoke-Checkliste #4 testet die Ownership-Ablehnung explizit per `curl PATCH … owner=inferred` — ohne das Feld wäre der Test nicht durchführbar (REST hätte nie eine niedrigere Priorität als `user` gesendet).
- [x] → Phase 3 (REST, Routing-Falle): **`{entity_id:path}` matcht Slashes** (IDs haben die Form `<type>/<slug>`), Starlette prüft Routen in Registrierungsreihenfolge und nimmt den ersten vollen (Pfad **und** Methode) Treffer. Eine generische `GET /entities/{entity_id:path}` **vor** `GET /entities/{entity_id:path}/lore` registriert hätte jede Lore-Anfrage stillschweigend als Entity-Lookup mit `entity_id="…/lore"` fehlinterpretiert (gleiches Risiko für `DELETE …/relationships` vs. `DELETE /entities/{id}`). Fix: alle Suffix-Routen (`/relationships`, `/lore`) stehen vor der bloßen `/entities/{id}`-Route. Per REST-Testfall verifiziert (`test_relationship_lifecycle_and_lore`) — ohne die Umsortierung wäre der Test mit falschem Handler-Treffer fehlgeschlagen.
- [x] → Phase 2/4 (Cache/Rebuild): **`EntityRepository.delete` löscht Kind-Zeilen explizit in Python**, nicht über DB-`ON DELETE CASCADE` — SQLite läuft in dieser App ohne `PRAGMA foreign_keys=ON` (`db/engine.py`), die deklarierten FKs erzwingen also nichts von selbst (gleiches Muster wie `api/collections.py::delete_collection`). Phase 4 (Rebuild) muss beim kompletten Cache-Neuaufbau entsprechend selbst leeren (`DELETE FROM knowledge_*` vor dem Reimport), nicht auf Cascade verlassen. — Eingearbeitet: `EntityRepository.clear_all()` leert alle vier `knowledge_*`-Tabellen in Kind→Eltern-Reihenfolge; `rebuild_cache` ruft es vor dem Reimport.
- [ ] → Phase 3 (Service): **Validator arbeitet zweigleisig.** `validate_metadata(dict, domain)` für nicht vertrauenswürdige Eingaben (Lesepfad, fängt „unbekannter Owner" ohne Parse-Crash), `validate_entity(entity, domain)` für typisierte Objekte (Schreibpfad). Beziehungstypen werden **gegen die Domäne** geprüft (nicht nur Entity-Typen) — der Service muss das im Blick behalten, wenn er Beziehungen anlegt.
- [ ] → Phase 3 (Service): **`vault.save_entity` ist reines I/O** (validiert + prüft Ownership NICHT). Der `KnowledgeService` muss vor jedem `save_entity` selbst validieren und die Ownership-Regel anwenden — Markdown-first.
- [x] → Phase 2/4 (Cache/Rebuild): **`vault.py` hat noch keinen Entity-Iterator.** Für Rebuild (Phase 4) fehlt ein `iter_entity_files()`/`load_all()` — bewusst nicht auf Vorrat gebaut (YAGNI), in Phase 4 ergänzen. Die Typ→Ordner-Zuordnung dafür kommt aus der Domäne (`folder_for`). — Eingearbeitet: `Vault.iter_entity_files()` (rglob `*.md`, überspringt `domains/`+`prompts/`) + `load_all()`. Beim Lesen ist keine Typ→Ordner-Zuordnung nötig — das geparste Frontmatter trägt `id`/`type`/`domain` selbst; `folder_for` braucht nur der Schreib-/Pfad-Pfad.

- [x] → Phase 4 (Entscheidung, Job-Anbindung): **Rebuild/Reconcile hängen am bestehenden Rebuild-Mechanismus, kein eigener `jobs/knowledge_*_job.py`.** Der Plan überließ die Wahl dem Bearbeiter (AK #4: „an vorhandenes Reconcile/Rebuild angehängt"). Umgesetzt: `RebuildTarget` in `jobs/rebuild_job.py` um `knowledge`/`knowledge_reconcile` erweitert, die Logik liegt testbar in `knowledge/maintenance.py`, der Job ist nur dünne async-Hülle (`_sync_knowledge`). Frontend nutzt den vorhandenen `triggerRebuild({target})`-Strang (null neue NgRx-Zeremonie, SSE-Fertig-Reset geerbt). **Folge für P27:** der reservierte Name `jobs/knowledge_*_job.py` bleibt frei für die *intelligenten* Jobs (Import/Update/Interview) — die Cache-Wartung ist bewusst keine solche.

- [ ] → P24+ (Reconcile-Genauigkeit, falls je nötig): **Reconcile hat keinen mtime-Vergleich — es re-importiert bei jedem Lauf alle Notizen.** `knowledge_entities` hat keine Zeitstempel-Spalte, also gibt es nichts zu vergleichen; voller Re-Import ist ein strikter Superset von „neuere Datei wird übernommen" und bei persönlicher Notizmenge vernachlässigbar. Wird der Vault je groß (tausende Entities) oder teuer zu parsen, eine `synced_at`/`mtime`-Spalte + inkrementellen Abgleich nachrüsten. **Löschung** entscheidet über Datei-*Existenz* (`_vault_file_exists`, Pfad allein aus `id`/`type`/`domain`), nicht über Import-Erfolg — eine vorhandene, aber defekt geparste Notiz behält ihre Cache-Zeile (kein Datenverlust aus einem Frontmatter-Tippfehler).
- [ ] → P24 (Packaging, falls relevant): **`domains/movies.yaml` wird per `Path(__file__).parent`-Relativpfad geladen**, nicht über `importlib.resources`. Läuft aus dem Source-Tree (uvicorn-Dev) einwandfrei; falls je ein installiertes Wheel ausgeliefert wird, muss die yaml-Datei explizit ins Wheel (`hatchling`-Include) und der Ladepfad ggf. auf `importlib.resources` umgestellt werden.
